# Chances 个人自用稳定版收尾计划

> **执行要求：** 使用 `executing-plans` skill 按任务逐项实施、验证和提交。

**目标：** 停止增加功能，把当前项目收敛成可长期稳定使用、可备份恢复、可诊断、可清晰启动的个人求职助手稳定版。

**架构：** 保留 FastAPI + SQLite + Streamlit + Chrome MV3 扩展。SQLite 继续作为唯一主数据源，Excel 仅用于导出，截图继续作为证据层。收尾阶段只允许修复缺陷、改善运行稳定性和补齐交付能力。

**技术栈：** Python 3.13、FastAPI、SQLite、Streamlit、Chrome Extension MV3、PowerShell、pytest。

---

## 一、冻结范围

### 本次必须完成

- 整理当前未提交代码，形成可追踪的稳定基线。
- 前后端一键启动、一键停止，并能明确报告启动失败原因。
- 数据库、截图和配置可以一键备份，并完成一次真实恢复演练。
- 中文界面、接口连接状态、空状态和错误提示统一。
- 用真实求职流程做端到端验收。
- 完善健康检查、README、故障排查和版本记录。
- 全量测试、启动测试和手工验收全部通过后发布稳定版本。

### 明确不做

- 不增加新的招聘网站。
- 不研究绕过 BOSS、拉勾等平台的反爬。
- 不接 OCR 或多模态视觉模型。
- 不开发 Word/PDF 简历导出。
- 不接邮件、日历和外部提醒。
- 不更换 Streamlit，不迁移到 React。
- 不增加新的评分、材料或数据分析功能。

---

## 二、发布标准

只有同时满足以下条件，项目才能标记为“个人自用稳定版”：

1. 双击或执行一个命令，可以稳定启动前端和后端。
2. 前端能明确显示 API 已连接或连接失败，不能再呈现误导性的空白页。
3. 重启电脑后，可以按文档在 2 分钟内恢复运行。
4. `data/jobs.db` 可以备份、校验并恢复。
5. 真实导入一个岗位后，可以完成审查、评分、准备、状态更新和复盘。
6. Chrome 扩展失败时有明确中文错误，不静默失败。
7. 所有自动化测试通过。
8. `doctor.py` 核心检查全部通过；可选 BOSS CLI 缺失不能导致主流程失败。
9. README 与实际运行方式一致。
10. 工作区整理完毕，稳定版本有明确提交和版本标签。

### 里程碑 2：稳定启动与停止

**目的：** 彻底解决“前端开了但 API 没开”“启动器双击没反应”“用户打开了 8000 接口而不是 8501 页面”等问题。

**新增文件：**

- `scripts/start_chances.ps1`
- `scripts/stop_chances.ps1`
- `scripts/status_chances.ps1`
- `tests/scripts/test_runtime_scripts.py`

**修改文件：**

- `README.md`
- `.streamlit/config.toml`
- 桌面启动器只作为仓库脚本的薄入口

**任务：**

1. 启动脚本检查虚拟环境、依赖、数据库目录和端口占用。
2. 先启动 FastAPI，轮询健康接口成功后再启动 Streamlit。
3. 正式启动不使用 `--reload`。
4. 保存进程号和日志路径。
5. 启动完成后明确输出前端地址 `http://127.0.0.1:8501`。
6. 停止脚本只停止本项目记录的进程，不能误杀其他 Python 进程。
7. 状态脚本报告前端、后端、数据库和端口状态。
8. Streamlit 使用访客工具栏模式，减少开发工具栏干扰。

**验收：**

- 连续执行三次“启动 -> 检查 -> 停止”均成功。
- 后端启动失败时，前端不会被继续拉起。
- 启动日志为中文或清晰可理解的英文技术日志。

---

### 里程碑 3：数据安全和恢复

**目的：** 让真实求职数据不会因为升级、误操作或数据库损坏而丢失。

**新增文件：**

- `scripts/backup_chances.py`
- `scripts/restore_chances.py`
- `tests/scripts/test_backup_restore.py`

**修改文件：**

- `scripts/doctor.py`
- `README.md`
- `.gitignore`

**备份范围：**

- `data/jobs.db`
- `data/captures/`
- 必要的本地配置
- 备份清单和生成时间

**任务：**

1. 备份前执行 SQLite `PRAGMA integrity_check`。
2. 使用 SQLite 安全备份 API，而不是直接复制正在使用的数据库。
3. 备份目录使用时间戳命名。
4. 恢复前自动备份当前数据。
5. 恢复后再次执行完整性检查和核心表检查。
6. 做一次真实恢复演练，并记录结果。

**验收：**

- 从备份恢复后，岗位数、画像、任务、事件和截图引用保持一致。
- 恢复失败不会覆盖当前有效数据库。

### 里程碑 5：真实端到端验收

**目的：** 测试的不只是函数，而是一次完整的真实求职操作。

**新增文件：**

- `docs/acceptance/personal-workflow-checklist.md`

**验收流程：**

1. 一键启动系统。
2. 打开 `http://127.0.0.1:8501`。
3. 确认个人画像正确。
4. 用扩展导入一个真实岗位，保留截图证据。
5. 在导入审查区接受岗位。
6. 在岗位池找到该岗位。
7. 查看岗位正文、截图、匹配建议和评分。
8. 生成准备包。
9. 更新一次状态和下一步动作。
10. 新建并完成一个跟进任务。
11. 查看周复盘是否包含这次变化。
12. 导出 Excel 并确认字段正确。
13. 重启系统，确认所有数据仍在。

**验收：**

- 整条流程无阻断。
- 每一步的结果能在下一步看到。
- 失败时有可理解的提示和恢复路径。

## 四、建议工期

按当前项目状态，建议用 4 个开发批次完成：

1. **批次一：** 基线整理、启动停止脚本、运行状态检查。
2. **批次二：** 数据备份恢复、诊断增强。
3. **批次三：** 前端错误状态和中文收口、真实端到端验收。
4. **批次四：** 文档、最终回归、版本提交与标签。

每个批次结束都必须保持测试通过，不能等到最后统一修复。

## 五、完成后的产品定义

完成本计划后，Chances 可以称为：

> **个人自用稳定版求职助手**

它不承诺自动绕过招聘网站反爬，也不追求继续增加功能；它承诺的是：

- 数据保存在本地并可以恢复。
- 每天能够稳定启动和使用。
- 岗位可以被导入、判断、准备、推进和复盘。
- 出现故障时能够诊断并恢复。
- 后续只进行缺陷修复和必要维护。
