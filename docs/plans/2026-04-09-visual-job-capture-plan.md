# 视觉化岗位采集改造方案

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将当前脆弱的 DOM/文本解析采集链路，升级为“截图优先、统一提要、结构字段辅助”的视觉采集方案，显著提升网页所见内容与系统入库内容的一致性。

**Architecture:** 保留现有浏览器扩展、FastAPI、SQLite 和 Streamlit 架构，不推翻已有项目。扩展端新增“截图采集”能力，优先保存当前岗位页面的视觉快照；后端新增视觉采集接口与截图存储；岗位入库改为“结构字段 + 视觉提要 + 原始截图引用”三层数据。DOM 解析不再作为主数据源，只作为补充和回退。

**Tech Stack:** Chrome Extension Manifest V3、FastAPI、SQLite、Streamlit、本地文件存储、可选 OCR/视觉模型接口

---

## 一、为什么要改

当前问题不是“个别字段偶尔不准”，而是采集策略本身有结构性缺陷：

- 页面真实看到的 JD 在主内容区，但现有方案会混入广告、导航、按钮文案、公司侧栏、推荐位。
- 不同站点 DOM 结构波动大，选择器一旦失效，就会出现标题、正文、公司字段错位。
- `document.body.innerText` 对长页面尤其不稳定，前后顺序、分段结构、隐藏文本都会被污染。
- 对你来说，最重要的是“页面真实表达的岗位内容”，而不是“爬到了一堆不可靠字段”。

因此新方案的核心原则是：

- 先记录你眼睛看到的页面证据
- 再基于页面证据生成统一文字提要
- 结构字段只做辅助，不再做真相来源

## 二、推荐的新主链路

### 方案结论

采用 **“视觉截图主链路 + DOM 辅助字段 + 人工确认”**。

### 新流程

1. 用户打开岗位详情页
2. 扩展触发截图采集
3. 扩展保存当前页一张或多张截图
4. 扩展同时采集最小结构字段
   - URL
   - 页面标题
   - 平台
   - 当前可识别到的岗位名 / 公司名 / 薪资 / 地点
5. 后端接收截图与结构字段
6. 后端生成“视觉提要任务”
7. 系统输出统一提要
   - 岗位摘要
   - 任职要求摘要
   - 公司信息摘要
   - 风险提示
8. SQLite 保存：
   - 原始结构字段
   - 视觉提要
   - 截图文件路径
   - 提要生成时间
9. 前端详情页优先展示视觉提要和截图

## 三、推荐的截图策略

### 推荐方案：分段截图

不建议第一版就做复杂的整页长截图拼接，优先做“关键区域分段截图”。

建议固定抓三类画面：

1. 顶部摘要区
   - 岗位名称
   - 薪资
   - 城市
   - 经验
   - 学历
   - 福利标签

2. 职位描述区
   - 岗位职责
   - 任职要求
   - 技术关键词

3. 公司信息区
   - 公司名称
   - 融资阶段
   - 公司规模
   - 行业

### 为什么先做分段截图

- 成本低，开发快
- 不需要第一阶段解决滚动拼接
- 更容易聚焦关键信息
- 后续即使接视觉模型，也更容易稳定提取

### 第二阶段再考虑

- 长页面滚动截图
- 自动拼接整页长图
- 对页面关键模块自动定位截图

## 四、数据模型改造建议

在现有 `jobs` 主表基础上，不重写，只增量扩展。

### jobs 表新增字段

- `capture_mode`
  - `dom`
  - `visual`
  - `hybrid`
- `hero_image_path`
- `description_image_path`
- `company_image_path`
- `visual_summary`
- `visual_summary_status`
  - `pending`
  - `ready`
  - `failed`
- `visual_summary_updated_at`
- `raw_capture_title`

### 新增 job_capture_assets 表

用途：一个岗位可以挂多张截图，后续也方便扩展 PDF、OCR 文本、页面快照。

建议字段：

- `id`
- `job_id`
- `asset_type`
  - `hero`
  - `description`
  - `company`
  - `fullpage`
- `file_path`
- `mime_type`
- `width`
- `height`
- `created_at`

### 新增 job_visual_summaries 表

如果你想把“原始岗位数据”和“AI/视觉提要结果”拆开，这张表更清晰。

建议字段：

- `requirements`

- `created_at`

## 五、扩展端改造方案

### 保留的部分

- `extension/content.js`
- `extension/popup.js`
- 当前平台识别逻辑
- 当前最小字段抓取逻辑

### 要改的部分

#### 1. 新增截图能力

优先使用：

- `chrome.tabs.captureVisibleTab`

第一版只抓当前可见区域，然后通过内容脚本自动滚到关键模块分别截图。

#### 2. 内容脚本新增关键区域定位

内容脚本新增：

- 定位顶部摘要区
- 定位职位描述区
- 定位公司信息区
- 滚动到区域顶部
- 每个区域通知 popup 执行截图

#### 3. popup 改成两段式采集

当前：

- 发送 `capture-page`
- 获取结构字段
- 直接调 `/api/import-page`

新流程：

1. 获取当前 tab
2. 请求 content script 返回结构字段和截图锚点信息
3. popup 依次让页面滚动到指定区域
4. 每个区域调用 `captureVisibleTab`
5. 将截图 base64 和结构字段一起发送到后端

### manifest 需要调整

- 增加必要权限说明
- 如需后台常驻处理，可考虑加 `background.service_worker`
- 第一版如果 popup 足够稳定，可以先不引入后台 worker

## 六、后端改造方案

### 新接口建议

保留现有 `/api/import-page`，但新增视觉接口，避免打断当前链路。

#### `POST /api/import-visual-page`

请求体建议：

- `url`
- `title`
- `platform`
- `extracted_job`
- `screenshots`
  - `hero`
  - `description`
  - `company`

返回：

- `job_id`
- `result`
- `capture_mode`
- `saved_assets`

### 文件存储建议

新增目录：

- `data/captures/<job_id>/hero.png`
- `data/captures/<job_id>/description.png`
- `data/captures/<job_id>/company.png`

如果岗位还没生成 `job_id`，可先用时间戳或 URL hash 临时目录，再落库后重命名。

### 后端视觉提要职责

第一版不要把“截图提要生成”和“截图保存入库”耦死。

建议拆两步：

1. 保存截图 + 入库岗位
2. 再生成视觉提要

好处：

- 即使提要失败，截图证据还在
- 用户能先看到原图
- 后续可异步重试提要

## 七、视觉提要生成策略

### 第一版建议

不要求一步到位做复杂 OCR 引擎，先做统一摘要框架。

输入：

- 顶部截图
- JD 区截图
- 公司信息截图
- 辅助字段

输出固定结构：

- `岗位摘要`
- `核心职责`
- `任职要求`
- `公司概况`
- `福利与亮点`
- `风险提示`

### 生成优先级

1. 先保存截图
2. 有能力时做 OCR/视觉理解
3. 没有视觉能力时，先用已有结构字段 + 标题生成占位摘要

### 关键原则

- 视觉提要是“用户阅读辅助”
- 原图始终保留
- 原图优先于摘要
- 摘要永远不是唯一真相

## 八、前端改造方案

### 首页

新增一个指标：

- 已有视觉截图的岗位数

### 岗位详情页

岗位详情要从“字段表”升级成“证据页”。

建议展示顺序：

1. 顶部结构摘要
2. 视觉提要
3. 岗位截图
4. 公司截图
5. 原始字段
6. 时间线

### 岗位池

新增字段：

- 采集方式
- 是否已有截图
- 是否已有视觉提要

### 操作按钮

新增：

- 重新生成视觉提要
- 查看原始截图

## 九、与现有方案的关系

### 保留

- FastAPI
- SQLite
- Streamlit
- 浏览器扩展
- 现有入库和状态流转

### 降级

- DOM 精确解析不再是主链路
- 站点选择器不再承担“最终 JD 还原”的责任

### 新定位

- DOM 解析：辅助字段
- 视觉截图：核心证据
- 视觉提要：统一阅读层

## 十、推荐迭代顺序

### 第一阶段：视觉证据先落地

目标：

- 扩展能抓三张关键截图
- 后端能保存截图
- 岗位能关联截图
- 前端能展示截图

完成标准：

- 你能在系统里看到岗位原始截图
- 即使字段不准，也能回看页面证据

### 第二阶段：视觉提要生成

目标：

- 根据截图生成统一摘要
- 详情页展示摘要

完成标准：

- 同一岗位的摘要结构固定
- 不再依赖站点文案结构差异

### 第三阶段：混合抽取优化

目标：

- 用 DOM 补足简单字段
- 用视觉提要补正文语义
- 两者联合输出

完成标准：

- 标题、薪资、城市等结构字段更稳
- JD 摘要更接近页面真实内容

## 十一、风险与取舍

### 风险 1：截图只拍到当前可视区域

解决：

- 第一版做分段截图
- 第二版再做整页滚动拼接

### 风险 2：视觉提要依赖模型能力

解决：

- 先把截图保存和展示独立落地
- 提要生成单独作为第二阶段

### 风险 3：图片存储体积上涨

解决：

- 图片压缩
- 只保留关键截图
- 后续做清理策略

### 风险 4：扩展操作时间更长

解决：

- 保持人工一键触发
- 加状态提示
- 第一版控制在 2-4 张图以内

## 十二、最终建议

建议正式把当前采集方案定义为：

- **主链路：视觉截图采集**
- **辅助链路：结构字段抓取**
- **展示层：统一文字提要 + 原始截图证据**

这比继续堆 Boss / 拉勾 / 猎聘 / 智联的 DOM 规则更符合你的真实目标，因为你要的不是“看起来采到了”，而是“系统里保存的内容和网页上看到的内容尽量一致”。

## 十三、实施建议

推荐下一步直接进入实现，按以下顺序最稳：

1. 扩展新增截图采集
2. 后端新增 `/api/import-visual-page`
3. SQLite 增加截图字段
4. 前端详情页展示截图
5. 再接视觉提要生成
